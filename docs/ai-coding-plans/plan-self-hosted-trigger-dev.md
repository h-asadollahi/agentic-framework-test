# Plan: Self-Hosted Trigger.dev (Fully Local, No Cloud)

## Context

The project currently references `TRIGGER_SECRET_KEY=tr_dev_...` in `.env.example` and the usage guide mentions a Trigger.dev cloud account. The user wants to run Trigger.dev **entirely locally** — no cloud platform dependency.

Trigger.dev is open-source (Apache 2.0) and fully self-hostable via Docker. The v4 Docker setup is streamlined — just `docker compose up` with no custom scripts.

## What changes

### 1. Update `.env.example`

Replace the cloud-oriented `TRIGGER_SECRET_KEY` with self-hosted config:

```bash
# Trigger.dev (self-hosted)
TRIGGER_API_URL=http://localhost:3040
TRIGGER_SECRET_KEY=tr_dev_...  # generated by the self-hosted instance
```

**File**: `.env.example`

### 2. Update `docs/usage-guide.md`

In **Section 1 (Prerequisites)** and **Section 2 (Running the System)**:

- Remove "A Trigger.dev account (free tier works for dev)" from requirements
- Add **Docker** to requirements
- Add a new section on setting up self-hosted Trigger.dev via Docker
- Update the run instructions to start Docker first, then dev servers
- Update `TRIGGER_SECRET_KEY` explanation

In **Section 11 (Production Considerations)**:

- Update the Trigger.dev deployment section to focus on self-hosted

### 3. Add `docs/trigger-dev-self-hosted.md`

A dedicated guide for the self-hosted setup:

- Docker prerequisites
- Clone the `triggerdotdev/docker` repo
- Run `./start.sh` (or `docker compose up`)
- Access dashboard at `http://localhost:3040`
- Create a project, get the secret key
- Configure `TRIGGER_API_URL` and `TRIGGER_SECRET_KEY` in `.env`
- Run `npm run trigger:dev` pointed at the local instance

### 4. Update `trigger.config.ts`

No code change needed — the SDK reads `TRIGGER_API_URL` from env automatically. The `project` field in `trigger.config.ts` will match whatever project ID is created in the local dashboard.
